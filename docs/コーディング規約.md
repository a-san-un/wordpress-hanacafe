# 📋 HanaCAFE nappa69 WordPressコーディング規約＆環境定義

> **SSOT**: データ構造・テンプレート階層・FLOCSS設計概要は [HanaCAFE nappa69 プロジェクト仕様書.md](HanaCAFE%20nappa69%20%E3%83%97%E3%83%AD%E3%82%B8%E3%82%A7%E3%82%AF%E3%83%88%E4%BB%95%E6%A7%98%E6%9B%B8.md) を参照すること。本ドキュメントはコーディング規約・環境定義・セキュリティ作法の詳細を定義する。

## 1. プロジェクト概要と技術スタック

- **言語設定**: 回答およびコメントは原則として**日本語**とする。
- **基本スタイル**:
  - **インデント**:
    - 全ファイル共通: **スペース2**（タブ禁止）
    - 対象: `.php` / `.js` / `.scss` / `.json` / `.yml` / `.md`
  - **波括弧**:
    - `.php`: **K&R（1TBS）**（Intelephense の formatOnSave に委任）
    - `.js` / `.scss`: **1TBS**（Prettier標準）
  - **フォーマットツール**: PHP は Intelephense、JS / SCSS / JSON / MD は Prettier
  - **命名規則**: 関数・変数はスネークケース、クラスはパスカルケースとする。
- **DB環境**: DB名 `hanacafe_db`, 接頭辞 `hc_`
- **CMS**: WordPress (バージョン 6.9+)
- **CSSメタ言語**: Dart Sass (最新仕様 `@use` 準拠)
- **CSS設計**: FLOCSS + BEMベース (モバイルファースト設計)
- **SCSSファイル命名規則**:
  - `layout/` レイヤー: `_l-` プレフィックス必須（例: `_l-header.scss`, `_l-footer.scss`）
  - `object/component/` レイヤー: `_c-` プレフィックス必須
  - `object/project/` レイヤー: `_p-` プレフィックス必須
  - layoutファイルにprojectレイヤーのスタイルを同居させない（例: `.p-drawer` は `_p-drawer.scss` で管理）
- **JavaScript**: Vanilla JS (モダンES6+)（jQuery依存は STEP 5-9 で完全撤去済み）
- **主要プラグイン**: Advanced Custom Fields (ACF) PRO相当のフィールド設計
- **カスタム投稿タイプ**: `menu` (メニュー管理)
- **MV画像管理**: CPT `main-visual` は廃止。ACF `pic` フィールド（固定ページ直接設定）で管理する。

## 2. 実務的インラインコメントとクリンネス規約

- **SCSSのサイレントコメント**: 開発メモや設計意図はすべて `//` で記述する。コンパイル後の `app.css` に `/* */` を出力させず、軽量化とクリーンさを保つ。
- **PHP/HTMLの隠蔽コメント**:
  - 画面上に文字として露出させないよう `<?php // ... ?>` を多用する。
  - 実装意図（「エスケープ処理の理由」「ACF取得意図」等）は PHPブロック内で記述し、ブラウザの「ソースを表示」に実装者のメモを露出させない。
- **物理削除の原則 (Physical Deletion)**:
  - コメントアウトによる「コードの残骸」をリポジトリに残さない。
  - 不要になった古いテンプレートタグ（`site-main`等）やデバッグ用コードは、コミット前に必ず物理的に削除する。
- **SCSSの随時クリーンアップ**:
  - PHP側でHTML構造（クラス名や要素）を変更・削除した場合、それに紐づくSCSSのスタイル定義（例：`&__no-img` 等）は、負債化を防ぐため即座に「物理削除」する。

## 3. WordPress テンプレート・ロジック規約

### ① マスターデータの取得

- **固定ページ参照**: 店舗情報（`access-info`）、共通設定（`common-info`）、メニュー設定（`menu-info`）など、役割ごとのマスターデータは必ずスラッグを指定して取得する。
- **推奨コード**: `get_hanacafe_master_page_id('スラッグ') ?: 0` を使用し、IDを動的に特定することで環境依存（IDの不一致）を排除する。
- **非推奨**: `get_page_by_path()` の直接使用は禁止。プロジェクト全域でラッパー関数（`get_hanacafe_master_page_id()`）に統一済み。

### ② テンプレートパーツの階層化

- **命名**: `template-parts/` 配下に、`home/`, `common/` などのディレクトリを切り、役割を明確にする。
- **読み込み**: `get_template_part()` を使用し、変数を渡す必要がある場合は第3引数の `$args` を活用する。

### ③ カスタム投稿タイプの登録 (functions.php)

- **規約**: `register_post_type` 時は、必ず `show_in_rest => true`（ブロックエディタ対応）と `supports` 配列の明示を行うこと。
- **必須要素**: `title`（alt属性等への流用）、`thumbnail`（画像用）を含めること。
- **補足**: `menu` 投稿タイプの実装では `supports` は `title`, `editor`, `thumbnail`, `custom-fields`, `revisions` を使用しており、`page-attributes` は含めない。

### ④ 防御的プログラミングの徹底

- **正規表現による操作**: HTML属性（`alt`, `class` 等）をプログラムで操作・置換する場合、クォーテーションの揺らぎ（`\"` と `'`）を許容するため、単純な文字列置換ではなく `preg_match` / `preg_replace` による正規表現を用いた実装を強く推奨する。

## 4. セキュリティとエスケープ処理

### ① 出力時のエスケープ徹底

- **原則**: データベースや外部関数から取得した値を出力する際は、直前で必ずエスケープ関数を通す。
- **使い分け**:
  - `esc_html()`: 一般的なテキスト出力。
  - `esc_url()`: リンク先 `href` や画像パス `src`。
  - `esc_attr()`: `alt` 属性、`class` 名、`id` 名などの属性値。特に動的な `alt` 属性には必須。

### ② ACF（Advanced Custom Fields）の扱い

- **取得**: `get_field()` を使用し、出力直前でエスケープする。
- **安全性**: 値が空である可能性を常に考慮し、`?:` (エルビス演算子) 等を用いて適切なフォールバック値を設定すること。

## 5. Git コミット規約（証左の構造化）

- **フォーマット**:
  - `feat:` (機能追加), `fix:` (バグ修正), `refactor:` (改善), `style:` (見た目・微修正)
- **構成**:
  1.  **概要**: 1行で要約。
  2.  **詳細**: 【バグ修正】【機能追加・変更】【ポリッシュ】の項目に分けて具体的に記述。
- **目的**: コミットメッセージ自体を、プロジェクトの健全性を証明する「証左」として機能させる。

## 6. テーマ設定 (theme.json) とスタイルの同期

### ① フォント定義の完全一致（SSOTの確保）

- **大原則**: エディタ側のフォント設定（`theme.json`）とフロントエンドの変数（`_variables.scss`）は、完全に同期していなければならない。
- **重複定義の禁止**: SCSS内で同じ変数名を複数回定義するような古いコードの残骸は、発見次第直ちに物理削除する。

### ② フォント変数の設計（\_variables.scss）

- **見出し用**: `$f-heading`（Montserrat + Noto Sans JP）
- **本文用**: `$f-body`（Montserrat + Noto Serif JP）
- `_variables.scss` と `theme.json` は常に同期させること。
